### 웹 서버

하드웨어 측면에서 웹 서버는 웹 서버 소프트웨어와 웹사이트 구성 요소 파일(HTML 문서, 이미지, CSS 스타일시트, JavaScript 파일 등)을 저장하는 컴퓨터입니다.

웹 서버는 인터넷에 연결되어 웹에 연결된 다른 장치와 물리적 데이터 교환을 지원합니다.

소프트웨어 측면에서 웹 서버는 웹 사용자가 호스팅된 파일에 액세스하는 방식을 제어하는 여러 부분으로 구성됩니다.

최소한 HTTP 서버가 포함됩니다. 

HTTP 서버는 URL(웹 주소)과 HTTP(브라우저가 웹 페이지를 보는데 사용하는 프로토콜)를 이해하는 소프트웨어입니다.

HTTP 서버는 저장된 웹사이트의 도메인 이름을 통해 액세스할 수 있으며, 호스팅된 웹사이트의 콘텐츠를 최종 사용자의 장치로 전달합니다.

브라우저가 웹 서버에 호스팅된 파일을 필요로 할 때마다 HTTP를 통해 파일을 요청합니다.

요청이 올바른 (하드웨어) 웹 서버에 도달하면 (소프트웨어) HTTP 서버는 요청을 수락하고, 요청된 문서를 찾아 HTTP를 통해 브라우저로 다시 보냅니다.

서버가 요청된 문서를 찾지 못하면 404 응답을 반환합니다. 

웹사이트를 게시하려면 정적 또는 동적 웹 서버가 필요합니다.

#### 정적 웹 서버

정적 웹 서버는 컴퓨터(하드웨어)와 HTTP 서버(소프트웨어)로 구성됩니다.

서버가 호스팅된 파일을 브라우저에 그대로 전송하기 때문에 "정적"이라고 합니다.

#### 동적 웹 서버

동적 웹 서버는 정적 웹 서버에 애플리케이션 서버와 데이터베이스와 같은 추가 소프트웨어가 추가된 것입니다. 

애플리케이션 서버가 HTTP 서버를 통해 브라우저로 콘텐츠를 보내기 전에 호스팅된 파일을 업데이트하기 때문에 "동적"이라고 합니다.

예를 들어 애플리케이션 서버는 데이터베이스의 콘텐츠로 HTML 템플릿을 채워 브라우저에 표시되는 최종 웹 페이지를 생성할 수 있습니다. 

MDN이나 위키백과와 같은 사이트는 수천 개의 웹 페이지를 가지고 있습니다.

일반적으로 이러한 종류의 사이트는 수천 개의 정적 HTML 문서가 아니라 몇 개의 HTML 템플릿과 거대한 데이터베이스로 구성됩니다.

이러한 설정을 통해 콘텐츠를 보다 쉽게 유지 관리하고 제공할 수 있습니다.

### 웹 서버의 작동 방식

#### 파일 호스팅

웹 서버는 모든 HTML 문서 및 관련 자산(이미지, CSS 스타일시트, JavaScript 파일, 글꼴, 비디오 포함)을 비롯한 웹사이트의 파일을 저장해야 합니다.

기술적으로는 이러한 파일을 모두 자신의 컴퓨터에 호스팅할 수 있지만, 다음과 같은 이유로 모든 파일을 전용 웹 서버에 저장하는 것이 훨씬 더 편리합니다. 

- 전용 웹 서버는 일반적으로 가용성(가동 및 실행)이 더 높습니다. 
- 다운타임 및 시스템 문제를 제외하고 전용 웹 서버는 항상 인터넷에 연결되어 있습니다. 
- 전용 웹 서버는 항상 동일한 IP 주소를 가질 수 있습니다. 이를 고정 IP 주소 라고 합니다. (모든 ISP가 홈 라인에 고정 IP 주소를 제공하는 것은 아닙니다.) 
- 전용 웹 서버는 일반적으로 제3자가 유지 관리합니다. 

이러한 이유로 우수한 호스팅 제공업체를 찾는 것이 웹사이트 구축의 핵심 요소입니다. 

#### HTTP를 통한 통신

웹 서버는 HTTP( H yper t ext T ransfer P rotocol)를 지원합니다.

이름에서 알 수 있듯이 HTTP는 두 컴퓨터 간에 하이퍼텍스트(링크된 웹 문서)를 전송하는 방법을 지정합니다.

프로토콜은 두 컴퓨터 간의 통신 규칙 집합입니다.

HTTP는 텍스트 기반의 상태 없는 프로토콜입니다. 

즉, 모든 명령은 일반 텍스트이며 사람이 읽을 수 있으며 서버와 클라이언트는 이전 통신을 기억하지 않습니다. 

예를 들어 HTTP만으로는 서버가 입력한 비밀번호나 완료되지 않은 트랜잭션의 진행 상황을 기억할 수 없습니다. 

이러한 작업에는 애플리케이션 서버가 필요합니다. 

HTTP는 클라이언트와 서버가 통신하는 방법에 대한 명확한 규칙을 제공합니다. 

- 일반적으로 클라이언트만 HTTP 요청을 하며, 서버에게만 요청합니다. 서버는 클라이언트의 HTTP 요청에 응답합니다. 서버는 서버 푸시라는 메커니즘을 통해 요청되기 전에 데이터를 클라이언트 캐시에 미리 채울 수도 있습니다. 
- HTTP를 통해 파일을 요청할 때 클라이언트는 파일의 URL을 제공해야 합니다.  
- 웹 서버는 모든 HTTP 요청에 최소한 오류 메시지로 응답해야합니다.  

웹 서버에서 HTTP 서버는 들어오는 요청을 처리하고 응답합니다.

HTTP 서버는 요청을 수신하면 요청된 URL이 기존 파일에 일치하는지 확인합니다.

일치하는 경우 웹 서버는 파일 콘텐츠를 브라우저로 다시 보냅니다.

일치하지 않는 경우 서버는 요청에 대해 동적으로 파일을 생성해야 하는지 확인합니다

이 두 가지 옵션이 모두 불가능한 경우 웹 서버는 브라우저에 오류 메시지(일반적으로 404 Not Found)를 반환합니다.

404 오류는 매우 흔하기 때문에 일부 웹 디자이너는 404 오류 페이지를 디자인하는 데 상당한 시간과 노력을 투자합니다.

#### 정적 콘텐츠 대 동적 콘텐츠

일반적으로 서버는 정적 콘텐츠 또는 동적 콘텐츠를 제공할 수 있습니다. 

정적 이라는 용어는 "있는 그대로 제공됨"을 의미합니다.

정적 웹사이트는 설정이 가장 간단하므로 첫 번째 사이트를 정적 사이트로 만드는 것이 좋습니다.

동적 이라는 용어는 서버가 콘텐츠를 처리하거나 데이터베이스에서 즉석에서 생성함을 의미합니다. 

이 방식은 유연성이 뛰어나지만 기술 스택이 더 복잡하여 웹사이트 구축이 훨씬 더 어려워집니다.

#### 마치며

모든 사용 사례에 적합한 기성 애플리케이션 서버는 없습니다.

일부 애플리케이션 서버는 블로그, 위키 또는 전자 상거래 솔루션을 호스팅하고 관리하도록 설계된 반면, 보다 일반적인 용도로 사용되는 서버도 있습니다.

대부분의 웹사이트 개발자는 기성 솔루션이 매우 많고 그 중 다수는 구성이 매우 용이하기 때문에 애플리케이션 서버를 처음부터 만들 필요가 없습니다. 

그러나 자체 서버를 만들어야 하는 경우 서버 프레임워크를 사용하여 기존 코드와 라이브러리를 활용하고 사용 사례를 충족하기 위해 필요한 부분만 확장하는 것이 좋습니다. 

임베디드 시스템의 엄격한 리소스 제약을 충족하는 등 비교적 소수의 개발자만 서버를 완전히 처음부터 개발해야 합니다.
