### HTTP란?

HTTP는 HTML 문서와 같은 리소스를 가져오는 데 사용되는 프로토콜로, 웹에서 데이터를 교환하는 기반이며 클라이언트-서버 프로토콜입니다.

즉, 일반적으로 웹 브라우저인 수신자가 요청을 시작합니다.

HTTP는 1990년대 초에 설계되었으며 시간이 지남에 따라 진화해 온 확장 가능한 프로토콜입니다.

HTTP는 TCP나 TLS로 암호화된 TCP 연결을 통해 전송되는 애플리케이션 계층 프로토콜이지만, 이론적으로는 모든 안정적인 전송 프로토콜을 사용할 수 있습니다.

#### HTTP 기반 시스템

HTTP 기반 시스템은 사용자 에이전트(클라이언트), 웹 서버, 프록시로 구성되어 있습니다.

 사용자 에이전트(일반적으로 웹 브라우저)는 사용자를 대신하여 요청을 보내고 서버는 요청된 문서를 제공합니다. 
 
 프록시는 클라이언트와 서버 사이에 위치하며 캐싱, 필터링, 로드 밸런싱 등 다양한 작업을 수행합니다.

#### HTTP의 기본 사항

HTTP는 단순하고 사람이 읽을 수 있으며 확장 가능하고 상태 비저장적이지만 세션리스하지 않으며 연결 기반입니다.

HTTP 메시지는 사람이 읽고 이해할 수 있어 개발자가 쉽게 테스트하고 초보자가 쉽게 이해할 수 있습니다.

HTTP 헤더를 사용하면 프로토콜을 쉽게 확장하고 실험할 수 있습니다.

HTTP는 상태 비저장적이지만 HTTP 쿠키를 사용하면 상태 저장 세션을 사용할 수 있습니다.

HTTP는 기본적으로 연결 기반인 TCP 표준을 사용합니다.

#### HTTP를 통해 제어할 수 있는 기능

HTTP를 사용하면 캐싱, 출처 제한 완화(Relaxing the origin constraint), 인증, 프록시 및 터널링, 세션과 같은 기능을 제어할 수 있습니다.

서버는 HTTP 헤더를 통해 프록시 및 클라이언트에 무엇을 얼마나 오래 캐싱할지 지시할 수 있습니다.

HTTP 헤더를 사용하면 서버 측에서 엄격한 출처 제한을 완화하여 

문서가 여러 출처의 정보를 모아놓은 형태가 될 수 있습니다.

HTTP는 WWW-Authenticate 및 유사 헤더를 사용하거나 HTTP 쿠키를 사용하여 특정 세션을 설정하여 기본 인증을 제공할 수 있습니다.

HTTP 요청은 네트워크 장벽을 통과하기 위해 프록시를 통해 전달될 수 있습니다.

HTTP 쿠키를 사용하면 요청을 서버의 상태와 연결하여 세션을 만들 수 있습니다.

#### HTTP 흐름

HTTP 흐름은 TCP 연결 열기, HTTP 메시지 보내기, 서버에서 보낸 응답 읽기, 연결 닫기 또는 재사용의 단계를 따릅니다.

HTTP/2에서는 HTTP 메시지가 프레임이라는 이진 구조에 포함되어 헤더 압축 및 다중화와 같은 최적화가 가능합니다.

#### HTTP 메시지

HTTP 메시지에는 요청과 응답의 두 가지 유형이 있으며 각각 고유한 형식이 있습니다.

요청에는 HTTP 메서드, 리소스 경로, HTTP 프로토콜 버전, 선택적 헤더, 본문이 포함됩니다.

응답에는 HTTP 프로토콜 버전, 상태 코드, 상태 메시지, HTTP 헤더, 본문(선택 사항)이 포함됩니다.

#### HTTP 기반 API

HTTP 기반 API 중 가장 일반적으로 사용되는 API는 Fetch API와 Server-Sent Events입니다.

Fetch API는 XMLHttpRequest API를 대체하여 JavaScript에서 HTTP 요청을 수행하는 데 사용됩니다.

Server-Sent Events는 서버가 HTTP를 전송 메커니즘으로 사용하여 클라이언트에 이벤트를 보낼 수 있는 단방향 서비스입니다.

#### 결론

HTTP는 사용하기 쉬운 확장 가능한 프로토콜입니다.

클라이언트-서버 구조와 헤더를 추가할 수 있는 기능을 통해 HTTP는 웹의 확장된 기능과 함께 발전할 수 있습니다.

HTTP/2는 성능 향상을 위해 HTTP 메시지를 프레임에 포함시켜 복잡성을 더했지만 메시지의 기본 구조는 HTTP/1.0 이후 동일하게 유지되었습니다.

세션 흐름은 기본적으로 유지되므로 HTTP 네트워크 모니터를 사용하여 조사하고 디버깅할 수 있습니다.
